Programe

整理在项目开发中好的习惯和思想。

代码开发

  1. 学会使用结构体将功能进行模块的，一个功能一个模块，简单明了易开发，比如frp的service中各种结构体，代表这个各种功能模块

  2. 经常需要使用的操作，只需要执行一次，然后想办法传递下去使用，可以放在结构体中，比如frp中的db

  3. 重复的代码需要进行抽象，减少代码繁重，抽象最好一步到位，比如frp中的db这个interface，第一步抽象就将表抽象出来了，抽象不彻底，代码繁重，现在就可以完全表的那一层都不需要，减少了大量的代码

  4. 模块一定要考虑优雅的退出

  5. 不要做多余的抽象，反而会增加代码量，比如frp中的每个表的crud，直接创建结构体就好，没有必要在之前做一层抽象，并不能起到很好的作用，包括后面的新增修改，都是一样的修改，这边做了一层抽象，反而会在修改的时候修改这一层代码。

  6. web服务的正常返回和错误返回可以单独抽象出一个对应函数给使用的时候直接调用，比如frp中的httputil。针对错误正常都是定义好的几种直接单独封装使用，比frp中的定义

var ( ErrRequest = NewError(http.StatusBadRequest, “bad request”) ErrRequestBody = NewError(http.StatusBadRequest, “bad request body”) ErrInternal = NewError(http.StatusInternalServerError, “internal error”) ErrUnauthorized = NewError(http.StatusUnauthorized, “unauthorized”) ErrForbiden = NewError(http.StatusForbidden, “operation forbidden”) ErrNotFound = NewError(http.StatusNotFound, “resource not found”) ErrLoginFailed = NewError(http.StatusUnauthorized, “invalid user or password”) )

7.处理数据库的时候需要加上上下文，设置超时这些数据。

  1. 代码格式很重要，直接影响代码的可读性，正常go代码直接使用go fmt处理一下 go fmt ./…

9.

整体架构

  1. 高并发需要注意的事情

*高并发下一定要减少锁的使用，这边也是channel在go中的重要作用之一。

*好的习惯是，稍大的类型存到map都存储指针而不是值。

*定义数据结构的时候，减少后面使用的转换

*defer是性能杀手，我的原则是能不用尽量避开。

*能不在循环内部做的，就不要在循环内存处理

*减少内存的频繁分配，减少使用全局锁的可能
